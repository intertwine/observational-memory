## Settled design decisions (grill round, 2026-05-28)

1. **#67 fail-closed behavior:** When `om context` is unavailable/fails, hooks emit **no additional agent context** and write a **one-line diagnostic to stderr** (e.g. `observational-memory: om context unavailable — run om doctor`). No raw `cat` of `profile.md`/`active.md`/`reflections.md`/`observations.md`. Keeps agent context clean while leaving a visible signal in logs.

2. **#70 output cap:** **Prompt ceiling + configurable char cap + post-hoc section-boundary trim.** Add `OM_REFLECTOR_OUTPUT_MAX_CHARS` (generous default), tighten the reflector prompt with an explicit section/line budget, and trim the result at a section boundary as a safety net if it overruns. Defer true streaming hard-stop unless this proves insufficient.

3. **#65 default budget:** **Raise the default input ceiling so it no longer binds below `OM_REFLECTOR_CONTEXT_MAX_CHARS`.** Today `_MAX_INPUT_TOKENS = 12_000` (~42k chars) silently clamps the configured 48k char reflections cap to ~12k effective. Promote the hardcoded ceiling and the `0.6` chunk ratio to env vars, derive/raise the default so the configured cap actually binds, and make diagnostics report both configured and effective values. Fixes the silent clamp for everyone, not just opt-in users.

4. **Workflow autonomy:** Per issue — implement in a worktree → validate (ruff + ruff format --check + pytest) → fresh-context adversarial review subagent → fix findings → re-validate → open a **DRAFT PR** + write a draft article. **STOP there.** Bryan does the review comments, walkthrough, sign-off, and merge. No version bump/publish.

## Per-issue acceptance criteria

### #67 — Fail closed on startup hook fallbacks
- `hooks/claude/session-start.sh` → `om context --for claude --cwd "$PWD"` only.
- `hooks/grok/session-start.sh` → `om context --for grok --cwd "$PWD"` only.
- `cowork_plugin/hooks/scripts/session-start.sh` → `om context --for cowork --cwd "$PWD"` only.
- Remove all raw `cat profile.md|active.md|reflections.md|observations.md` fallback blocks.
- On `om context` failure/missing `om`: emit nothing to context, one-line stderr diagnostic, exit cleanly.
- Tests simulate `om context` failure / missing `om` and assert generated-memory contents are NOT emitted.
- Maintainer doc note so future hooks don't reintroduce unbounded generated-file fallbacks.

### #65 — Reflector budget knobs + honest diagnostics
- Promote `_MAX_INPUT_TOKENS` (reflect.py:23) → `OM_REFLECTOR_MAX_INPUT_TOKENS`; promote the `0.6` observation-chunk ratio (reflect.py:412, :515) → `OM_REFLECTOR_OBSERVATION_CHUNK_RATIO`.
- Default input ceiling raised so it does not bind below `OM_REFLECTOR_CONTEXT_MAX_CHARS` for the current corpus shape.
- Replace the misleading `exceeds OM_REFLECTOR_CONTEXT_MAX_CHARS=12143` warning with one reporting **configured vs effective** caps + the binding ceiling (e.g. `configured_reflections_cap=48000 effective_reflections_cap=12143 max_input_tokens=12000 observation_chunk_budget=25200`).
- Test: configured reflections cap higher than the effective per-call cap is surfaced honestly.
- A 2x-corpus scenario can reflect without dropping most of `reflections.md`.
- Documented in `docs/configuration.md`.

### #70 — Cap reflect output on the Codex path
- `OM_REFLECTOR_OUTPUT_MAX_CHARS` (generous default) — works on the Codex path (no dependency on the rejected `max_output_tokens`).
- Reflector prompt carries an explicit section/line budget.
- Post-call trim lands on a section boundary (never mid-section); warns when it fires.
- Test exercises the cap on the Codex path (mocked Responses).
- Documented in `docs/configuration.md` alongside other reflector knobs.

### #63 — Async Batch error UX
- Wrap `submit_reflect_batch` (the `POST /batches` call + preceding file upload) in `_reflect_async`: catch `openai.APIStatusError`/`RateLimitError`/`BadRequestError` → `click.ClickException` with an actionable message (e.g. `OpenAI Batch submission failed: billing hard limit reached. Check your OpenAI plan/limits (project-scoped keys have per-project spend caps).`), non-zero exit, no stack trace.
- Same for the synchronous fallback path when the LLM call fails after retries (llm.py:127 retry-exhausted `RuntimeError`).
- Non-retryable 400 still NOT retried; only 429 retried (unchanged).
- Failed submit still leaves no dangling job record and writes nothing (unchanged — add a regression test).

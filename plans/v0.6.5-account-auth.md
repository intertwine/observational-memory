# v0.6.5 — ChatGPT and xAI account login for observer / reflector

Status: proposed
Owner: Bryan
Target release: v0.6.5
Related issue: https://github.com/intertwine/observational-memory/issues/51 (token budgets — independent)

## Problem

`om observe` and `om reflect` call OpenAI or Anthropic via direct API keys
(`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`). At higher-tier models (GPT-5, GPT-5.5,
Claude Opus) per-call cost adds up fast across hooks, schedulers, and
reflection passes, and the only opt-in we offer today is "set a cheaper
default model".

Both target ecosystems publish a *subscription-backed* path that bypasses the
metered API tier and is included in plans most power users already pay for:

- **OpenAI ChatGPT** (Plus / Pro / Team / Enterprise) via the Codex OAuth
  device-code flow against `https://auth.openai.com`, routed to the Codex
  backend `https://chatgpt.com/backend-api/codex`.
- **xAI Grok** (SuperGrok subscription) via the OIDC authorization-code +
  PKCE **loopback** flow against `https://auth.x.ai`. This is the same flow
  upstream Hermes (`nousresearch/hermes-agent` `hermes_cli/auth.py`,
  provider id `xai-oauth`) shipped on 2026-05-23.

v0.6.5 makes both flows first-class, available to **any** `om` user with no
dependency on having Codex CLI or Grok CLI installed.

## Goals

1. `om login openai-chatgpt` runs a complete OAuth device-code flow against
   `auth.openai.com` (port of Hermes's Codex flow). `om login xai-oauth` runs
   a complete authorization-code + PKCE loopback flow against `auth.x.ai`
   (port of Hermes's new xAI flow). Both work for any user on a fresh
   machine, with no other CLI prerequisites.
2. Refresh tokens automatically when access tokens are within a skew window of
   expiry; never block scheduled jobs on user interaction.
3. `om` owns its own auth store. We never write to `~/.codex/` or `~/.grok/`.
4. Convenience: `om login --import` opportunistically reads `~/.codex/auth.json`
   and `~/.grok/auth.json` for users who already have those CLIs and want a
   zero-prompt setup. Not the primary path.
5. Existing API-key flows (`anthropic`, `openai`) and direct xAI API-key flow
   stay supported and behave identically to today.
6. `om install` and `om doctor` surface the new options clearly so a new user
   discovers the cheap path before they discover their first OpenAI bill.

## Non-goals

- Token budgeting and per-call cost tracking (#51).
- Anthropic Claude Pro / Max subscription auth — Anthropic does not currently
  publish a third-party-client inference endpoint.
- Cluster-syncing tokens. Auth state stays host-local.
- A general-purpose OAuth framework — we ship only the two flows we use.

## Upstream evidence (read today, 2026-05-23)

Upstream Hermes (`nousresearch/hermes-agent` `hermes_cli/auth.py`, commit
`5fd3676`, pushed 2026-05-23) implements both providers. v0.6.5 ports their
shape and security hardening as-is — they have a docs page at
https://hermes-agent.nousresearch.com/docs/guides/xai-oauth-oauth.

**OpenAI ChatGPT — device-code flow, against Codex endpoints:**

```
issuer:               https://auth.openai.com
device-code request:  POST https://auth.openai.com/api/accounts/deviceauth/usercode
device-code poll:     POST https://auth.openai.com/api/accounts/deviceauth/token
token exchange:       POST https://auth.openai.com/oauth/token  (grant_type=authorization_code)
refresh:              POST https://auth.openai.com/oauth/token  (grant_type=refresh_token)
client_id:            app_EMoamEEZ73f0CkXaXp7hrann       (Codex CLI)
inference base URL:   https://chatgpt.com/backend-api/codex
auth_mode tag:        chatgpt
```

**xAI Grok — authorization-code + PKCE loopback flow, against discovery URLs:**

```
issuer:                       https://auth.x.ai
discovery:                    GET https://auth.x.ai/.well-known/openid-configuration
authorization_endpoint:       (from discovery; pinned to *.x.ai)
token_endpoint:               (from discovery; pinned to *.x.ai)
client_id:                    b1a00492-073a-47ea-816f-4c329264a828   (Grok CLI; reused, see below)
scope:                        openid profile email offline_access grok-cli:access api:access
redirect_uri:                 http://127.0.0.1:56121/callback        (loopback, RFC 8252)
PKCE:                         S256, verifier echoed at token step (xAI server quirk)
extra authorize params:       plan=generic   (without this, accounts.x.ai rejects loopback OAuth
                                              from non-allowlisted clients)
                              referrer=observational-memory   (xAI server-log attribution)
inference base URL:           https://api.x.ai/v1   (pinned to api.x.ai / *.x.ai)
auth_mode tag:                oidc
refresh skew:                 120s (matches Hermes XAI_ACCESS_TOKEN_REFRESH_SKEW_SECONDS)
SSH/headless mode:            manual-paste fallback prints authorize URL + reads callback URL from stdin
tier-denied detection:        HTTP 403 from token endpoint → recommend XAI_API_KEY + `provider: xai`
```

Why loopback authorization-code instead of device-code for xAI: upstream
Hermes confirmed via #26990 that xAI's device-code path has rough edges; the
loopback flow is what they ship. RFC 8252 §7 endorses loopback for native
apps, and OAuth-correctness is enforced via state + nonce + PKCE.

## Client-id strategy

Both providers require an OIDC `client_id`. v0.6.5 follows Hermes's precedent
and reuses the first-party CLI client ids, since neither provider currently
publishes a third-party registration portal:

| Provider           | Client id we use                                | Source                |
|--------------------|-------------------------------------------------|-----------------------|
| `openai-chatgpt`   | `app_EMoamEEZ73f0CkXaXp7hrann`                  | Codex CLI (Hermes uses the same) |
| `xai-oauth`        | `b1a00492-073a-47ea-816f-4c329264a828`          | Grok CLI  (Hermes uses the same) |

Both ids are env-overrideable (`OM_OPENAI_CHATGPT_CLIENT_ID`,
`OM_XAI_OAUTH_CLIENT_ID`) so a user or packager who registers their own
client can swap them without a code change. The Hermes xAI flow has been
hardened around this exact client id (e.g. the `plan=generic` workaround for
the `accounts.x.ai` allowlist) — porting that behaviour means we inherit a
known-good configuration.

A v0.7 follow-up tracks asking xAI and OpenAI to mint first-party
`observational-memory` clients.

## Design

### New providers

Two new values for `OM_LLM_PROVIDER`, plus a cleaner direct-key xAI provider:

| Provider id        | Transport / endpoint                                                            | Auth                                  | Default model         |
|--------------------|----------------------------------------------------------------------------------|---------------------------------------|-----------------------|
| `openai-chatgpt`   | OpenAI Chat Completions, `base_url=https://chatgpt.com/backend-api/codex`         | OAuth (subscription)                  | `gpt-5-codex`         |
| `xai-oauth`         | OpenAI-compatible chat, `base_url=https://api.x.ai/v1` (Bearer access token)      | OIDC device-code (subscription)       | `grok-4.3`            |
| `xai` (new)        | OpenAI-compatible chat, `base_url=https://api.x.ai/v1`                            | `XAI_API_KEY` (metered fallback)      | `grok-4.3`            |

The OpenAI SDK accepts `base_url=` and an arbitrary `api_key=` so the
subscription access token "looks like" an API key to the SDK — exactly how
Hermes routes Codex traffic today.

`auto`-mode resolution order (new entries in **bold**):

1. `anthropic` if `ANTHROPIC_API_KEY` present.
2. `openai` if `OPENAI_API_KEY` present.
3. **`openai-chatgpt`** if om has valid (refreshable) ChatGPT tokens.
4. **`xai-oauth`** if om has valid (refreshable) xAI tokens.
5. **`xai`** if `XAI_API_KEY` present.

Result: today's users see no change. New users discover account login via
`om install` / `om login` / `om doctor`.

### Auth store

New module: `src/observational_memory/auth/`

```
auth/
  __init__.py
  store.py             # JSON-on-disk auth store with file lock + 0600 perms
  pkce.py              # S256 verifier/challenge helpers
  remote.py            # SSH/remote-session detection (port of _is_remote_session)
  oauth_loopback.py    # loopback HTTP server, state+nonce+PKCE, manual-paste fallback
  oidc_discovery.py    # discovery fetch + host-pinning validator
  openai_chatgpt.py    # Codex-flavored device-code + refresh against auth.openai.com
  xai_oauth.py         # loopback authorization-code + refresh against auth.x.ai
  cli_import.py        # opportunistic readers for ~/.codex/auth.json, ~/.grok/auth.json
  commands.py          # `om login`, `om logout`, `om auth status` implementations
```

Storage path: `~/.config/observational-memory/auth.json` (same parent as the
existing env file). 0600 on POSIX; relies on NTFS ACL on Windows. File lock
via `fcntl` on POSIX, `msvcrt` on Windows — 10s timeout, same shape as Hermes.

### Auth flows

**OpenAI ChatGPT — device-code** (port from upstream Hermes
`_codex_device_code_login`, `refresh_codex_oauth_pure`,
`_codex_access_token_is_expiring`):

1. POST `https://auth.openai.com/api/accounts/deviceauth/usercode` with
   `client_id`. Receive `user_code` + `device_auth_id`.
2. Print the URL `https://auth.openai.com/codex/device` and the user code.
   Best-effort `webbrowser.open()`. Detect remote/SSH sessions and skip the
   browser hint.
3. Poll `https://auth.openai.com/api/accounts/deviceauth/token` every
   `interval` seconds. On 200, exchange via `POST
   https://auth.openai.com/oauth/token` with `grant_type=authorization_code`,
   `client_id`, `code_verifier`, `redirect_uri` returned by the device-auth
   exchange.
4. Persist `access_token`, `refresh_token`, `expires_at`, last refresh ISO.
5. Refresh: POST same `/oauth/token` with `grant_type=refresh_token`.

**xAI Grok — loopback authorization-code + PKCE** (port from upstream Hermes
`_xai_oauth_loopback_login`, `_xai_oauth_build_authorize_url`,
`_xai_oauth_exchange_code_for_tokens`, `refresh_xai_oauth_pure`):

1. Fetch and validate `https://auth.x.ai/.well-known/openid-configuration`.
   Validate that `authorization_endpoint` and `token_endpoint` hosts are
   `x.ai` or `*.x.ai` (anti-MITM, mirrors Hermes
   `_xai_validate_oauth_endpoint`).
2. Bind a local HTTP server on `127.0.0.1:56121/callback` (fall back to a
   nearby free port if 56121 is taken; honour the same range Hermes uses for
   troubleshooting parity).
3. Generate `code_verifier` (43–128 char URL-safe), `code_challenge =
   base64url(sha256(verifier))`, plus random `state` and `nonce`.
4. Build the authorize URL with these query params:
   `response_type=code`, `client_id`, `redirect_uri=http://127.0.0.1:<port>/callback`,
   `scope=openid profile email offline_access grok-cli:access api:access`,
   `code_challenge`, `code_challenge_method=S256`, `state`, `nonce`,
   `plan=generic`, `referrer=observational-memory`.
5. Open the authorize URL in a browser (or, if `_is_remote_session()` or
   `--no-browser` or `--manual-paste`, print it and read the resulting
   redirect URL from stdin — the Hermes "manual paste" flow).
6. Wait on the loopback server for the `?code=...&state=...` callback.
   Validate state. Render an OAuth-friendly HTML success page with a CORS
   header so xAI's success script can dismiss the consent screen
   (`_xai_callback_cors_origin` from Hermes).
7. POST the token endpoint with `grant_type=authorization_code`, `code`,
   `redirect_uri`, `client_id`, `code_verifier`, *and* echo
   `code_challenge` + `code_challenge_method` (Hermes's defense for the
   `code_challenge is required` quirk in xAI's token endpoint, see #26990).
8. Detect HTTP 403 → raise `xai_oauth_tier_denied` with a clear message
   recommending `XAI_API_KEY` + `OM_LLM_PROVIDER=xai` for users whose
   subscription tier doesn't cover OAuth.
9. Persist `access_token`, `refresh_token`, `expires_at`, discovery snapshot,
   redirect URI, last refresh ISO.
10. Refresh: POST same token endpoint (re-validated against `*.x.ai`) with
    `grant_type=refresh_token`, `client_id`, `refresh_token`. Same
    tier-denied handling.

A shared `auth/oauth_loopback.py` module hosts the loopback HTTP server,
state/nonce/PKCE helpers, browser-open + manual-paste + SSH-hint behaviour,
and CORS rendering. The xAI flow uses it directly; the OpenAI ChatGPT flow
does not (device-code) but shares the PKCE and remote-session helpers.

### LLM call path changes

`src/observational_memory/llm.py`:

1. Dispatcher gains `openai-chatgpt`, `xai-oauth`, `xai`. All three call a
   shared `_call_openai_compatible(base_url, api_key, model, ...)` helper.
2. Before constructing the client, call
   `auth.resolve_runtime_credentials(provider, config)` which:
   - reads the om auth store under lock,
   - if the access token is missing or within `_REFRESH_SKEW_SECONDS=120` of
     `expires_at`, calls the provider-specific refresh function,
   - persists refreshed tokens back to the store,
   - returns `(access_token, base_url)`.
3. Retry: existing `_is_retryable` logic unchanged. A `401` from a
   subscription-backed call triggers one forced refresh and a single retry
   before bubbling up.

### Subscription-cost semantics

`openai-chatgpt` and `xai-oauth` calls are zero-marginal-cost to the user. The
budget subsystem (#51) will tag them with `auth_mode in {chatgpt, oidc}` so
reports display `est_total_usd=0.00 (subscription)` instead of per-token
pricing. v0.6.5 only needs to record `auth_mode` on each call; budget
consumption is later work.

### Compatibility & safety rails

- API-key paths untouched. `OM_LLM_PROVIDER=anthropic` and
  `OM_LLM_PROVIDER=openai` behave identically to today.
- Auth store is host-local and never enters OM Cluster (matches existing
  precedent for `~/.codex/auth.json` and `~/.grok/auth.json`).
- We never write `~/.codex/auth.json` or `~/.grok/auth.json` — read-only on
  opt-in import.
- `om logout` removes only entries om wrote.
- Tokens never logged. CLI output redacts to last 4 characters.
- ToS notice printed on first `om login`: *"By logging in, you confirm `om`'s
  use of your ChatGPT / xAI subscription complies with the provider's terms of
  service."*

## Risks and mitigations

- **Client-id authorization drift**: providers can revoke the Codex / Grok CLI
  client ids' permissions. Mitigation: env-var overrides
  (`OM_OPENAI_CHATGPT_CLIENT_ID`, `OM_XAI_OAUTH_CLIENT_ID`); fall back to
  API-key providers; `om doctor` flags 403 errors with `restricted_client` /
  `invalid_client` codes.
- **xAI subscription tier denial**: SuperGrok base / lower tiers don't grant
  `grok-cli:access` via OAuth. Token endpoint returns HTTP 403 with a
  tier-specific code. Handled as `xai_oauth_tier_denied` with a clear error
  recommending `XAI_API_KEY` + `OM_LLM_PROVIDER=xai` and a link to upgrade
  at https://x.ai/grok. Mirrors Hermes upstream behaviour.
- **xAI inference base-URL exfiltration**: a malicious `XAI_BASE_URL` /
  `OM_XAI_OAUTH_BASE_URL` would ship the OAuth bearer to an attacker.
  Mitigation: pin base URL to `api.x.ai` or `*.x.ai`, refuse other hosts
  during config load with a logged warning. Mirrors Hermes
  `_xai_validate_inference_base_url`.
- **Discovery cache poisoning**: a tampered `/.well-known/openid-configuration`
  could redirect the token endpoint off-domain. Mitigation: validate both
  `authorization_endpoint` and `token_endpoint` against `x.ai` / `*.x.ai` on
  every fetch (and after re-reading from store). Mirrors Hermes
  `_xai_validate_oauth_endpoint`.
- **ChatGPT backend ToS**: covered by the precedent set by Hermes and Codex
  CLI; we show a one-time ToS notice on first login and document that this is
  the user's own subscription being used through their own client.
- **Token theft surface area**: auth store is 0600; tokens never logged;
  status output redacts to last 4 chars; tests assert no full token reaches
  stdout.
- **Hermes drift**: we vendor snapshots of both the Codex device-code flow
  and the xAI loopback flow from `nousresearch/hermes-agent`
  `hermes_cli/auth.py` (commit `5fd3676`, 2026-05-23). Cite the upstream
  file path + commit at the top of `auth/openai_chatgpt.py` and
  `auth/xai_oauth.py` so future maintainers can pull bug fixes (the xAI
  flow in particular has accreted several non-obvious workarounds for xAI
  server quirks — #26990 for the `code_challenge is required` quirk, the
  `plan=generic` allowlist workaround, the CORS handler for the consent
  screen — and we want to know where to look when the next one lands).

## Plan amendment (2026-05-23, during live E2E proof)

Live testing against real ChatGPT Plus/Pro and SuperGrok accounts surfaced
several places where the design above was wrong. Code was corrected to match
reality; this section records the deltas so the plan and code agree.

### 1. Subscription providers must be sticky in `_infer_provider`

The plan said `_infer_provider` learns `grok-*` and Codex prefixes. It did not
anticipate the interaction with a global `OM_LLM_MODEL` of a *different* family.
Bryan's env sets `OM_LLM_MODEL=gpt-5.5`; with `OM_LLM_PROVIDER=openai-chatgpt`,
the original `gpt-*` → `openai` rule silently rerouted calls to the **metered**
OpenAI API — exactly the bill v0.6.5 exists to avoid.

Fix: when the user explicitly selects `openai-chatgpt` or `xai-oauth`,
`_infer_provider` returns that provider unchanged. Model-name inference never
routes *away* from a chosen subscription provider to a metered one. Regression
test: `tests/auth/test_llm_routing.py`.

### 2. ChatGPT Codex backend is the Responses API, not Chat Completions

The plan (and the provider table) assumed `chatgpt.com/backend-api/codex` is a
drop-in OpenAI **Chat Completions** endpoint and that "the token just looks like
an API key to the SDK." Live testing disproved this in four steps:

1. `POST .../chat/completions` returns a **Cloudflare challenge page** unless the
   request advertises a first-party originator. Fixed by porting Hermes's
   `_codex_cloudflare_headers` (`agent/auxiliary_client.py`): `originator:
   codex_cli_rs`, a `codex_cli_rs/...` `User-Agent`, and `ChatGPT-Account-ID`
   from the JWT `chatgpt_account_id` claim. → `auth/openai_chatgpt.cloudflare_headers`.
2. With Cloudflare cleared, `/chat/completions` returns **HTTP 404** — the Codex
   backend speaks the **Responses API** (`/responses`), like codex-rs. Switched
   the `openai-chatgpt` path to `client.responses.create(...)`.
3. The Responses call must use `input=[{role, content:[{type:input_text,...}]}]`
   (not a bare string → 400), `store=False` (→ 400), and `stream=True` (→ 400).
4. `max_output_tokens` is rejected as an unsupported parameter → not forwarded.

These are implemented in `llm._call_codex_responses`. The xAI path is unaffected:
`api.x.ai/v1` is a genuine OpenAI-compatible Chat Completions endpoint and works
through the shared `_call_openai_compatible` helper.

### 3. ChatGPT-account Codex model allow-list shifts; default is now `gpt-5.5`

`gpt-5-codex` (the plan's default) is rejected with HTTP 400 "model is not
supported when using Codex with a ChatGPT account." The live
`GET .../backend-api/codex/models` endpoint listed `gpt-5.5`, `gpt-5.4`,
`gpt-5.4-mini`, `gpt-5.3-codex`, `gpt-5.3-codex-spark`, `gpt-5.2`. Default
changed to `gpt-5.5` (current flagship); `OM_OPENAI_CHATGPT_MODEL` overrides it
when the allow-list moves again (it has drifted repeatedly per Hermes notes).

produced real model output to `observations.md` via xAI's then-current default
(SuperGrok, +37 lines) and `gpt-5.5` (ChatGPT Codex Responses, +240 lines).

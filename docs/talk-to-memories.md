# Talk to your memories

`om talk` lets you have a back-and-forth conversation with your own memory. Each
time you say something, Om runs **recall over your memories in the background**,
then answers grounded in what it found. Today the conversation is text-based;
the same loop is built to take a real voice transport (microphone + speech) as a
drop-in later.

```bash
om talk
```

```
om talk — recall backend 'bm25' is ready.
Talking to your memories. Type 'exit' or press Ctrl-D to end.
you> what was I working on last week?
om> You were building the section-targeted reflection feature and shipping v0.7.0.
  ⟢ grounded in 3 memory snippet(s)
you> exit
```

## How it works

```
you speak ─▶ background recall over memories ─▶ grounded reply ─▶ Om speaks
                 (Moss / bm25 / qmd backend)        (your LLM)
```

- **Recall** uses whatever search backend you've configured (`OM_SEARCH_BACKEND`).
  It runs off the conversation's critical path so listening never blocks.
- **The reply** is generated by your configured LLM provider — the same one
  `om reflect` uses. Replies are kept short and plain so they read aloud well.
- **Grounding:** the recalled snippets are handed to the model with each turn.
  If recall finds nothing relevant, Om says so instead of making things up.

If the backend isn't ready (for example, you've never built an index), the
conversation still works — it just runs ungrounded. Build the index first with:

```bash
om talk --reindex          # rebuild the index, then start talking
```

## Options

| Flag | What it does |
| --- | --- |
| `--query "..."` | An opening line to start the conversation. |
| `--backend NAME` | Recall backend for this session: `moss`, `bm25`, `qmd`, `qmd-hybrid`, `none`. Defaults to `OM_SEARCH_BACKEND`. |
| `--limit N` | Max memory snippets recalled per turn (default 5). |
| `--max-turns N` | Stop after N turns. |
| `--reindex` | Rebuild the search index before talking. |
| `--for AGENT` | Host agent name for profile/recall routing (e.g. `claude`). |
| `--json` | Print the full transcript as JSON when the conversation ends. |

## Using Moss for fast recall (optional, cloud)

[Moss](https://www.moss.dev) is a real-time semantic-search runtime. With Moss
as the backend, recall is semantic (meaning-based, not just keywords) and very
fast once the index is loaded into memory.

**Moss is cloud-backed and opt-in.** Building the index **uploads your memory
text** to the Moss cloud (`service.usemoss.dev`). Om never enables it for you.
To turn it on:

```bash
pip install "observational-memory[voice]"   # installs the moss SDK

export OM_SEARCH_BACKEND=moss
export OM_MOSS_PROJECT_ID=your-project-id
export OM_MOSS_PROJECT_KEY=your-project-key   # secret — keep it out of git

om talk --reindex                             # uploads memory, builds the index, then talks
```

Privacy notes:

- Memory sections tagged `scope=local` are **withheld** from the upload, the
  same rule that keeps host-local entries out of shared cluster memory.
- Your project key is a secret. Om never logs or prints it.
- Prefer `bm25` (the default) if you want recall to stay fully on your machine
  with zero upload. `om talk` works exactly the same way with it.

Optional tuning:

- `OM_MOSS_INDEX_NAME` — index name (default `observational-memory`).
- `OM_MOSS_MODEL_ID` — embedding model; blank uses the SDK default (`moss-minilm`).
- `OM_MOSS_ALPHA` — hybrid keyword/vector blend; blank uses the SDK default.

## Voice (planned)

The transport layer is pluggable: `om talk` talks through a `VoiceTransport`,
and the default `TextTransport` reads and writes text. A microphone + speech
transport (speech-to-text in, text-to-speech out) plugs into the same loop
without changing the recall or conversation code. It is intentionally not in
this release because it needs audio hardware that can't be tested in CI.
